Editions & Roadmap

版本路线

Version Plan当前路线分为三个阶段：1.0 做稳交付与执行，1.1 完成试用到续费闭环，2.0 扩展标准化协作与更多能力接入。

不同阶段对应不同使用方式与升级路径。

SmartFlow 当前的版本节奏可以这样理解： 1.0 先把真实交付、可信执行和本地接入做稳， 1.1 把自动注册、支付、续费和低客服闭环补起来， 2.0 再决定哪些部分值得继续产品化。

当前 1.0 的核心定位是：智流是一个以 Python 脚本为第一种成熟执行资产的多租户能力治理与执行平台。

1.0 可信交付1.1 低客服闭环2.0 下一步收敛持续升级按需扩展

1.0

可信 AI 执行交付版

  • 一个客户一个租户
  • 统一能力目录、脚本接入与 Skill 映射
  • 当前以 Python 脚本作为第一种成熟执行资产
  • 智能大脑推荐、确认、执行、审计
  • 模板分发、安装、回滚、日志
  • 免登录体验页、登录 / 注册页、个人技能库 / 上传 / 公告 / 个人空间 / 社区广场，以及 smartflow-cli 最小闭环

Next

下一阶段先收敛这些

  • 把自动注册、试用开通、支付、续费、到期提醒和停开流程补成低客服闭环
  • 把小团队协作里真正高频的能力收成更稳的方案
  • 把套餐、额度和交付边界收敛成更清楚的产品规则
  • 把 CLI / 本地执行里值得产品化的部分继续补稳

1.0 Core

1.0 当前重点

  • 以 Python 脚本为第一种成熟执行资产
  • Skill 继续作为用户可见的能力单元
  • 交付、执行、安全边界和持续更新形成闭环
  • 单能力执行已经可用
  • 本地桥接与 CLI 支撑更深接入

2.0 Expansion

2.0 继续扩展什么

  • 自动注册、支付、续费、提示和自助支持进一步产品化
  • 更通用的能力资产注册模型
  • API、数据连接器、文档模板、审批流程等更多能力接入
  • 安装、分发、升级和回滚策略进一步通用化
  • 小团队协作、套餐化和标准产品能力继续收敛

Upgrade Route

从免费入口到正式交付，版本阶段怎么承接

01. 免费入口感知价值

先通过公开体验页、登录 / 注册页和免费入口理解平台是什么；登录后的个人入口当前会继续落到技能库、上传、公告、个人空间和社区广场，确认“AI 调度能力”这条链路是否成立。

02. CLI 与更深使用开始接入

更深一层的用户会继续看到 smartflow-cli、本地脚本接入和混合执行桥梁能力，这些已经属于当前 1.0 的一部分。

03. 企业 1.0 正式交付承接收入

当客户需要独立网址、独立租户、正式权限和交付验收时，企业 1.0 单租户交付就是当前最适合的路径。

04. 2.0 才把标准方案继续做厚

2.0 会继续扩展小团队协作、套餐化、资源计量和更完整的产品梯度。

Current Upgrade Logic

升级路径怎么走

  • 第一步先让用户通过免费入口感知价值
  • 第二步用 CLI、客户流程和部署文档承接更深问题
  • 第三步当用户进入真实客户场景时，由企业 1.0 或小团队方案承接
  • 第四步先把 1.1 的自动支付、续费和提示做成低客服闭环，再进入 2.0 的标准产品梯度

What 2.0 Changes

2.0 之后，这条路径会发生什么变化

  • 免费入口、小团队方案和正式交付之间的边界会更明确
  • 套餐、配额、资源、转化和运营指标会收敛得更清楚
  • CLI + 本地 Agent 里真正有价值的部分会继续往稳定方案靠拢
  • “先体验，再按场景升级”的路径才会真正收成标准化链路

Near-Term Focus

当前阶段的标准服务包

企业 AI 工作流搭建

把现有脚本、审批和执行确认链收成正式交付包，优先服务中小团队的真实自动化场景。

自动化运维中枢

围绕任务调度、日志留痕、模板分发和本地执行桥接，形成最容易成交和复用的标准化服务包。

行业数据自动化交付包

把高频行业数据采集、清洗、导出和报告链路沉淀成窄场景交付包，再逐步进入 2.0 的标准产品化。

Why Both Lines

为什么企业线和免费版要并行

  • 只做企业线：增长依赖人工接单，用户反馈样本不足
  • 只做免费线：交付场景不足，现金流不足
  • 所以当前路线保持为：企业交付 1.0 承接收入，个人免费版承接入口，两条线共用同一套可信执行底座

Current Rhythm

当前版本节奏

  • 1.0 是当前已经有可用基础的交付主线
  • 1.1 优先把自动注册、支付、续费、提醒和低客服支持做稳
  • 2.0 是下一阶段的产品化收敛
  • 更多能力会按需求逐步扩展

Next Read

如果你已经理解版本路线，下一步通常不是继续聊概念

更有效的做法通常是回到当前 1.0 和 1.1 的真实交付链路，把文件场景、交付物、接入准备和自动化运营边界一次对齐，再决定是否要继续走 2.0 的产品化升级路径。
