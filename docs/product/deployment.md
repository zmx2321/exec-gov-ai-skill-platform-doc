# 部署方式

SmartFlow 的部署思路不是只有一种，而是根据数据边界、客户规模和执行位置做组合。

## 1. 单租户交付模式

适合：

- 企业客户项目交付
- 需要独立访问地址
- 需要和其他客户完全隔离

特点：

- 一个客户一个租户
- 客户能力与数据边界清晰
- 适合长期维护和持续扩展

## 2. 平台治理模式

适合：

- 平台方统一管理多个客户
- 需要公告、模板、日志和台账总控
- 需要跨租户治理但不打破隔离

特点：

- `public` 作为平台治理域
- 租户保持独立运行
- 平台级能力和租户级能力分流

## 3. 免费入口模式

适合：

- 产品体验入口
- 个人开发者试用
- 公开展示平台能力

特点：

- 入口语义更轻
- 更强调上手速度和体验路径
- 为后续团队版、企业版升级提供前置入口

## 4. 本地混合执行模式

适合：

- 文件和数据不能全部上云
- 任务依赖本地环境或内网资源
- 需要保留本地执行灵活性

特点：

- 平台负责入口、规划、权限和审计
- 本地环境负责执行
- CLI / Agent 负责桥接

## 5. 未来边缘与设备模式

这是 `4.0` 的公开方向之一。

适合：

- 需要把执行端从脚本环境延伸到边缘节点
- 需要统一治理设备能力包
- 需要对高风险动作做权限和审计控制

特点：

- 能力单元不再局限于脚本
- 执行端可能是本地 Agent、边缘节点或设备插件
- Web 控制面继续作为统一治理入口

## 6. 如何选择

如果你当前重点是：

- 客户交付：优先单租户交付模式
- 平台运营：加上平台治理模式
- 获客与试用：加上免费入口模式
- 私有数据执行：补本地混合执行模式
- 长期设备化延伸：预留边缘与设备模式

SmartFlow 的价值不在于强迫所有用户走同一条部署路径，而在于用同一套治理逻辑覆盖不同执行位置。
